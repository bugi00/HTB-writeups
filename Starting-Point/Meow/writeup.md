# Meow - Hack The Box Writeup

## 1. 개요

Machine: Meow  
Difficulty: Very Easy  
Operating System: Linux  

이 문제는 Telnet 서비스의 잘못된 인증 설정을 이용하여 원격 시스템에 접근하는 문제이다.  
핵심은 열린 서비스가 무엇인지 확인한 뒤, 해당 서비스의 특성을 바탕으로 가장 가능성 높은 접근 방식을 빠르게 선택하는 것이다.

이 문제에서 중요한 것은 단순히 접속에 성공하는 것이 아니라,  
왜 Telnet에 주목했고 왜 root 계정으로 바로 시도했는지를 이해하는 것이다.

---

## 2. Enumeration

대상 시스템의 열린 포트와 실행 중인 서비스를 확인한다.

nmap -Pn -sC -sV <TARGET_IP>

![nmap](./images/nmap.png)

결과:

23/tcp open  telnet  

스캔 결과 23번 포트에서 Telnet 서비스가 실행 중인 것을 확인할 수 있다.

여기서 Telnet이 중요했던 이유는 다음과 같다.

- Telnet은 원격 접속을 위한 서비스이다.
- 서비스 자체가 셸 접근과 직접 연결될 가능성이 높다.
- 보안 수준이 낮고, 오래된 환경에서는 잘못된 인증 설정이 남아 있을 가능성이 있다.

즉, 웹 서비스나 파일 공유 서비스처럼 추가 탐색이 필요한 유형이 아니라,  
서비스가 열려 있는 순간 바로 인증 상태를 점검해볼 가치가 높은 상황이다.

따라서 다음 단계는 복잡한 추가 스캔이 아니라,  
Telnet에 직접 접속하여 인증이 정상적으로 적용되는지 확인하는 것이다.

---

## 3. Analysis

Telnet은 원격 접속을 위한 프로토콜이다.

주요 특징은 다음과 같다.

- 평문 통신을 사용한다.
- 일반적으로 사용자 인증이 필요하다.
- 설정이 잘못된 경우 인증이 매우 약하거나 사실상 없는 상태로 운영될 수 있다.

이 문제에서 Telnet이 보이자마자 바로 접속을 시도한 이유는,  
Telnet이 단순 정보 제공 서비스가 아니라 직접 시스템 접근으로 이어지는 서비스이기 때문이다.

또한 이 문제는 Starting Point 난이도이므로,  
복잡한 취약점 악용보다는 잘못된 기본 설정이나 인증 부재를 점검하는 방향이 더 합리적이다.

즉, 여기서의 판단 흐름은 다음과 같다.

1. 원격 접속 서비스가 열려 있다.
2. 해당 서비스는 잘못 설정되면 바로 시스템 접근으로 이어질 수 있다.
3. 따라서 가장 먼저 인증 우회 또는 인증 부재 가능성을 확인한다.

이 문제에서는 바로 그 판단이 맞았고,  
별도의 우회 기법 없이 직접 로그인 시도가 공격의 핵심이 된다.

---

## 4. Exploitation

Telnet 서비스를 이용하여 대상 시스템에 접속한다.

telnet <TARGET_IP>

![telnet login](./images/telnet-login.png)

접속 후 로그인 프롬프트가 출력된다.

login:

여기서 중요한 것은 “어떤 계정을 먼저 시도할 것인가”이다.

이 문제에서 root 계정을 먼저 시도한 이유는 다음과 같다.

- Linux 환경에서 가장 대표적인 기본 관리자 계정은 root이다.
- Starting Point 문제 특성상 복잡한 사용자 열거보다 대표 계정 점검이 우선이다.
- 만약 인증 설정이 부실하다면, 가장 위험한 계정이 바로 root일 가능성이 높다.

따라서 root 계정으로 로그인을 시도한다.

root

![telnet shell](./images/telnet-shell.png)

별도의 비밀번호 입력 없이 로그인에 성공하며 쉘에 접근할 수 있다.

이 시점에서 확인할 수 있는 사실은 다음과 같다.

- Telnet 서비스가 외부에 노출되어 있다.
- root 계정으로 로그인이 가능하다.
- 비밀번호 검증이 없거나 사실상 무력화되어 있다.
- 즉, 인증 없는 원격 관리자 접근이 가능한 상태이다.

이 문제의 공격은 복잡한 익스플로잇이 아니라,  
서비스 특성을 이해하고 가장 가능성 높은 약점을 바로 찌른 것에 가깝다.

---

## 5. Flag 획득

로그인 이후 현재 디렉토리의 파일을 확인한다.

ls

시스템 접근이 성공했으므로 다음 단계는 실제 목표물인 flag 파일의 위치를 확인하는 것이다.

Starting Point 문제에서는 보통 현재 위치 또는 접근 가능한 경로에 flag 파일이 존재하므로  
우선 현재 디렉토리의 파일 목록부터 확인하는 것이 가장 효율적이다.

flag 파일을 확인한 후 내용을 출력한다.

cat flag.txt

이를 통해 최종적으로 flag를 획득할 수 있다.

여기서 중요한 점은,  
flag 획득 자체보다 “인증 없는 원격 관리자 접근이 이미 성립했다”는 사실이 더 치명적이라는 것이다.  
flag는 단지 성공 여부를 확인하는 증거일 뿐이고,  
실제 보안상 문제는 root 권한 수준의 직접 접근이 가능했다는 데 있다.

---

## 6. Root Cause

이 문제의 근본 원인은 다음과 같다.

- Telnet 서비스가 외부에 노출되어 있다.
- root 계정에 대해 정상적인 인증 절차가 적용되지 않는다.
- 그 결과 비밀번호 없이 원격 로그인이 가능하다.

즉, 단순히 “Telnet을 사용했다”가 문제가 아니라,  
“Telnet이 외부에 열려 있고, 가장 민감한 계정에 대해 인증 통제가 사실상 없었다”는 점이 핵심이다.

이 설정은 공격자 입장에서 별도의 취약점 분석이나 우회 기법 없이도  
즉시 시스템 제어권 확보로 이어질 수 있으므로 매우 위험하다.

## 8. 결론

이 문제는 서비스 식별 이후,  
해당 서비스의 성격을 바탕으로 가장 위험한 시나리오를 우선 검증하는 것이 얼마나 중요한지 보여준다.

Telnet은 원격 접속 서비스이므로,  
열려 있는 것이 확인되면 단순 정보 수집보다 인증 상태 점검이 우선되어야 한다.

특히 이 문제에서는 다음 판단이 핵심이었다.

- 23번 포트에서 Telnet 서비스 확인
- Telnet은 직접 시스템 접근과 연결되는 서비스임
- Starting Point 환경에서는 잘못된 기본 인증 설정 가능성이 높음
- 따라서 root 계정으로 직접 로그인 시도
- 비밀번호 없이 쉘 접근 성공

결국 이 문제의 본질은 “취약한 서비스 설정”이다.  
보안성이 낮은 프로토콜을 외부에 노출하는 것 자체도 위험하지만,  
그 위에 인증 절차까지 부실하면 시스템 전체가 즉시 노출된다.

실제 환경에서는 다음과 같은 대응이 필요하다.

- Telnet 사용 금지 또는 최소화
- 외부 노출 차단
- root 직접 로그인 금지
- 정상적인 인증 정책 적용
- SSH와 같은 더 안전한 대체 수단 사용

이 문제는 기술적으로 매우 단순하지만,  
운영 설정 하나가 전체 시스템 보안 수준을 무너뜨릴 수 있다는 점에서 중요한 사례이다.
